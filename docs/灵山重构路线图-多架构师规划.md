## 🗺️ 四份架构报告关键结论交叉验证

| 维度 | Architect A（全景） | Architect B（后端） | Architect C（前端） | Architect D（数据） |
|------|-------------------|-------------------|-------------------|-------------------|
| **核心矛盾** | 赛题景区为灵山，数据层未切换 | 需新建 5 张表 + 11 端点 | 灵山配色已定，需替换 20+ 处前端文本 | 5 个知识库文件全量替换 |
| **工作量** | 3 阶段 ~12h | P0-P5 共 3h20min | 6 模块 ~6.5 天 | 5 文件 + 3 代码修改 |
| **最大风险** | VRM 降级不阻塞对话 | Streamlit 替换时机 | 移动端地图 Leaflet CDN 与 CSP | Dify 未部署时降级方案 |
| **不做建议** | 多景区切换 / 原生 App | 多语言 / 支付系统 | 小程序原生重构 | PostgreSQL 迁移 |

### 四份报告一致结论

> **"不重写、不替换、只做增量改造"** — 所有架构师均建议在现有架构上做 Edit 级修改，不引入新框架，不拆分大文件。

## 📐 技术方案总览

### 数据库：5 张新表（backend/repository/scenic_repo.py）

| 表名 | 内容 | 记录数 |
|------|------|--------|
| `scenic_area_info` | 景区基本信息（票务/电话/开放时间） | 1 行 |
| `scenic_spots` | 景点（名称/分类/评分/坐标/亮点） | 9 行 |
| `scenic_activities` | 活动（名称/时间/地点/标签） | 4 行 |
| `scenic_routes` | 路线（名称/时长/途经景点） | 3 行 |
| `scenic_route_spots` | 路线-景点关联 | 19 行 |

### REST API：11 个新端点（backend/routes/scenic.py）

| 方法 | 端点 | 用途 | 鉴权 |
|------|------|------|------|
| GET | `/api/scenic/area` | 景区信息 | 否 |
| GET | `/api/scenic/spots` | 景点列表（支持筛选/分页） | 否 |
| GET | `/api/scenic/spots/{id}` | 景点详情 | 否 |
| POST | `/api/scenic/spots` | 新增景点 | 需 |
| PUT | `/api/scenic/spots/{id}` | 更新景点 | 需 |
| DELETE | `/api/scenic/spots/{id}` | 删除景点 | 需 |
| GET | `/api/scenic/activities` | 活动列表 | 否 |
| GET | `/api/scenic/routes` | 路线列表 | 否 |
| GET | `/api/scenic/routes/{id}` | 路线详情（含途经景点） | 否 |
| GET | `/api/scenic/stats` | 大屏统计数据 | 否 |
| GET | `/api/scenic/ai-responses` | 预设 AI 回复 | 否 |

## 🚫 排除项（YAGNI — 本次不做）

| 排除功能 | 原因 |
|---------|------|
| ❌ 多景区切换（用户可选西湖/灵山） | 赛题只要求一个景区 |
| ❌ 微信小程序开发 | 7.6MB VRM 无法塞进 2MB 包 |
| ❌ 灵山 VRM 定制模型建模 | 当前模型可复用，新模型非评分点 |
| ❌ 替换 Streamlit 为 admin.html | 方案B 共存更安全 |
| ❌ 从 SQLite 迁移 PostgreSQL | 赛题不要求 |
| ❌ GPU 推理（Wav2Lip/SadTalker） | 纯 CPU 约束 |
| ❌ 用户注册登录系统 | 赛题不要求 |

## 📋 强制要求

1. **输出必须具体**：所有代码/配置/颜色写死具体值，不留占位符
2. **增量修改**：已有文件只做 Edit，不做 Write 覆盖
3. **降级优先**：每个外部依赖配降级方案（VRM→静态头像、DeepSeek→通义千问）
4. **Token 节省**：高频问题走预设回复缓存，跳过 LLM 调用
5. **测试护航**：每次修改后 `pytest tests/ -v` 通过
6. **Ruff 检查**：`ruff check .` 零 Error
7. **文件上限** ≤ 500 行，函数 ≤ 50 行，嵌套 ≤ 4 层
8. **不随手优化**：修 bug 不改无关代码，不加没要求的功能
9. **先跑通再打磨**：骨架→功能→样式→动效，逐步叠加
