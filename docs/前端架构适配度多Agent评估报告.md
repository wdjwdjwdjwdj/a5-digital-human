## 一、prototype-v2 前端重构集成评估

### 📊 集成可行性：**中**

### 📋 概要

prototype-v2 包含三个高质量的前端原型页面（admin.html 2178行、dashboard.html 1210行、mobile-prototype.html 2273行），均为纯前端 HTML/CSS/JS，设计精美但**景区绑定于无锡灵山胜境**（配色 #2B5F75 黛青/#C73E3A 朱红，9个灵山景点、4个灵山活动）。而当前项目基于**杭州西湖**（配色 #2d6a4f 翠绿 / #fefae0 宣纸 / #d4a373 琥珀），核心数据层面存在根本性不一致。

**核心矛盾**：三个原型页面零 API 集成（全部从全局变量 `LingshanData` 读取硬编码数据），将其改造为真实后端调用相当于重写大部分 JS 交互逻辑。

### ⚠️ 冲突点

1. **景区数据不一致** — spots.js 中 9 个景点、4 个活动、3 条路线全部是无锡灵山数据，无法直接用于西湖项目
2. **原型零 API 调用** — admin.html/dashboard.html/mobile-prototype.html 全部通过 `LingshanData` 全局变量读数据，无任何 fetch/axios 调用
3. **mobile-prototype.html 布局不可直接生产化** — 390x844px 固定 phone-frame 模拟器布局，非响应式设计
4. **VRM 数字人缺失** — 原型页面无 Three.js/VRM 集成，与当前核心功能的 3D 数字人展示不兼容
5. **配色主题冲突** — 灵山黛青配色与西湖水墨翠绿配色需要统一决策

### 🔧 需新增/修改的后端 API

| 端点 | 方法 | 用途 | 优先级 |
|------|------|------|--------|
| `/api/scenic/info` | GET | 景区基本信息 | P0 |
| `/api/scenic/spots` | GET | 景点列表（支持筛选/分页） | P0 |
| `/api/scenic/spots/{id}` | GET | 单个景点详情 | P0 |
| `/api/scenic/activities` | GET | 今日活动列表 | P0 |
| `/api/scenic/routes` | GET | 推荐路线 | P1 |
| `/api/scenic/dashboard` | GET | 大屏统计数据 | P1 |

需新建 `backend/repository/scenic_repo.py`（与现有 `chat_repo.py` 分离，符合单一职责原则）。

### 🎨 需改造的前端模块

1. **admin.html** — 所有 `LingshanData.spots` 引用改为 fetch 调用 `/api/scenic/*`
2. **dashboard.html** — 所有 `LingshanData.dashboard` 引用改为调用 `chat_repo` 统计 API
3. **mobile-prototype.html** — phone-frame 布局完全重写为响应式
4. **相对路径修正** — `src="data/spots.js"` 改为 `src="/prototype/data/spots.js"`（3 个文件批量替换）

### 🏢 Admin 后台策略

**推荐：替换 Streamlit 后台**

prototype-v2 的 admin.html 提供了更完整的 CRUD 体验（景点管理、活动管理、路线管理），这是 Streamlit 后台完全不具备的。但需将硬编码数据改为 API 调用，且需保留 Streamlit 的知识库管理功能（作为 API 暴露）。

| Leaflet (unpkg.com) | ❌ 未允许 | 需在 script-src + style-src 新增 |
| echarts-wordcloud (cdn.jsdelivr.net) | ✅ 已允许 | 无 |

## 二、移动端微信小程序改造评估

### 📊 可行性：**低（不建议）**

### 📋 概要

微信小程序与当前技术栈存在 **6 项根本性不兼容**：无 DOM/BOM API、WebGL 有限、包大小 2MB 限制无法容纳 7.6MB VRM 模型、不支持 SSE/ReadableStream、不支持 Web Audio API、不支持 importmap CDN。移除 3D VRM 数字人等于移除本项目的最大技术亮点。

### 🚫 关键约束

1. ❌ 小程序无 DOM/BOM（无 window/document/importmap）
2. ❌ 小程序 WebGL 有限（Three.js 不完全兼容）
3. ❌ 小程序包大小 ≤ 2MB（VRM 模型 7.6MB 超限）
4. ❌ 小程序不支持 SSE/ReadableStream（需改 WebSocket）
5. ❌ 小程序不支持 Web Audio API（需改 wx API）
6. ❌ 学生无企业资质（小程序发布受限）

### 🔧 后端适配（如果需要）

- 新增 WebSocket/WSS 端点替代 SSE
- TTS 音频格式调整为小程序兼容格式
- 域名配置微信白名单 + HTTPS 强制

### 方案对比

| 维度 | 方案A：原生重构 | 方案B：WebView 壳 | 方案C：增强 H5+PWA ✅ |
|------|:--------------:|:----------------:|:--------------------:|
| 工作量 | 20-25 人天 | 3-5 人天 | **6-8 人天** |
| 3D VRM 保留 | ❌ 移除 | ✅ 保留 | **✅ 保留** |
| SSE 保留 | ❌ 改 WebSocket | ✅ 保留 | **✅ 保留** |
| 代码复用率 | ~20% | ~95% | **~100%** |

| 技术风险 | 🔴 高 | 🟡 中 | **🟢 低** |

### 💡 建议

**强烈推荐方案C：不做小程序，改为增强 H5 响应式 + PWA。**

具体改造清单（6-8 人天）：

## 三、纯 HTML5 迁移评估

### 📊 当前状态：**✅ 已达标**

### 📋 概要

逐条验证 W3C 纯 HTML5 定义标准，**当前前端 100% 已属于纯 HTML5 应用**。

### ✅ 通过验证的模块

| 模块 | 技术 | 合规依据 |
|------|------|---------|
| 3D 数字人 | Three.js + importmap | importmap 是 HTML 规范标准部分 |
| TTS 播报 | Web Audio API | W3C 标准 API |
| ASR 识别 | Web Speech API | W3C 标准 API |
| 流式通信 | SSE + ReadableStream | HTML5 标准 |
| 字体 | Google Fonts CSS @import | 标准 CSS |
| 数据持久化 | localStorage | Web Storage 标准 |
| 图片上传 | FileReader + FormData | 标准 File API |

### 🔄 唯一非合规模块：Streamlit 管理后台

Streamlit 是 Python 服务端渲染框架，不符合纯 HTML5 定义。但：
- **独立进程**（端口 8501），与主前端解耦
- **内部管理工具**，非游客可见
- **推荐保留**：替换成本 3-5 天，收益为零

### 💡 建议

**保持现状，不做改造。** 纯 HTML5 已达标，唯一非合规的 Streamlit 是独立内部工具，不值得投入时间替换。应将精力集中在：

1. 问答准确率提升（目标 95%+）
2. 端到端延迟优化（目标 < 3s）
3. VRM 口型同步精度改进
4. 知识库覆盖度扩展

### 🔧 ECC 工作流影响

纯 HTML5 选型与 ECC 工作流 **无冲突**。仅 TypeScript 特定工具（tsc、TS 审查器）不适用，但 `code-reviewer`、`security-reviewer`、`performance-optimizer` 全部正常工作。
