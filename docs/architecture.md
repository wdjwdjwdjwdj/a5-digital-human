## 核心流程

1. 用户语音输入 → FunASR 转文字（降级: Web Speech API）
2. 文本 → Dify RAG 检索景区知识（降级: 直连 DeepSeek）
3. 检索结果 + 用户问题 → DeepSeek LLM 生成回答（降级: 通义千问）
4. 回答文本 → Edge-TTS 合成语音（降级: pyttsx3）
5. 语音播报 + VRM 3D 口型同步（降级: 静态 PNG + CSS）

## 技术选型理由

| 模块 | 选型 | 选型理由 |
|------|------|---------|
| LLM | DeepSeek API | 中文理解能力强，价格约 1 元/百万 token，性价比极高 |
| RAG | Dify 社区版 | 开源免费，Docker 一键部署，可视化知识库管理 |
| ASR | FunASR | 阿里达摩院开源，中文识别准确率最高，支持 CPU 推理 |
| TTS | Edge-TTS | 微软免费服务，中文自然度高，延迟低，无需 API Key |
| 数字人 | Three.js + @pixiv/three-vrm | WebGL 3D 渲染，VRM 1.0 标准，BlendShape 口型同步 |
| 管理后台 | Streamlit | Python 原生，无需前端框架，快速搭建管理界面 |
| 数据库 | SQLite | 零配置，Python 内置，适合单机部署场景 |

## 创新点

1. **三级降级链路**：Dify RAG → DeepSeek → 通义千问，确保任何环节故障不中断服务
2. **纯 CPU 运行**：零 GPU 依赖，降低部署门槛，适合景区普通服务器
3. **双模 ASR**：服务端 FunASR + 浏览器 Web Speech API 自动切换
4. **景区主题定制**：灵山配色系统（#2B5F75 / #F5F0E8 / #C73E3A）+ 景点快捷入口

## 数据流

- 对话记录持久化到 SQLite `conversations` 表
- 文档上传历史持久化到 SQLite `upload_history` 表
- TTS 音频文件存储在 `frontend/static/audio/`（运行时生成，不入 Git）

## 技术要点

- 纯 CPU 运行，零 GPU 依赖
- 每个外部依赖均有降级方案
- 对话记录持久化到 SQLite
- 代码质量：Ruff 零 Error，pytest 10/10 通过
